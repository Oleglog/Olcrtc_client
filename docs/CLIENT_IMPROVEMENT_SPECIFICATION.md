# Техническое задание на развитие olcRTC Client

Статус документа: проект ТЗ для следующего цикла разработки

Базовая версия: `Olcrtc_client v1.0.18`

Дата: 2026-07-17

## 1. Назначение документа

Это ТЗ описывает дальнейшее развитие независимого Android-клиента olcRTC после стабилизации версии `v1.0.18`.

Главные цели следующего цикла:

1. Уменьшить реальное и воспринимаемое время подключения, отключения и восстановления соединения.
2. Сделать lifecycle VPN предсказуемым при сворачивании приложения, выключении экрана, потере сети и переходах Wi-Fi ↔ мобильная сеть.
3. Сделать все длительные операции отменяемыми и не блокирующими управляющие команды.
4. Улучшить главный экран, редакторы, подписки, настройки и выбор приложений без полного переписывания приложения.
5. Сохранить уже подтверждённо рабочий datapath, импорт, подписки, безопасность и release pipeline.

ТЗ не требует немедленно реализовать все пункты одним релизом. Работы должны выпускаться небольшими проверяемыми этапами, чтобы сетевые изменения не смешивались с большим редизайном.

## 2. Подтверждённая исходная точка

По результатам проверки пользователя `v1.0.18` работает корректно:

- WBStream и Telemost подключаются;
- браузер, YouTube, Telegram и другие приложения получают доступ в интернет через VPN;
- приложение продолжает обслуживать VPN в фоне;
- ручные подключения и подключения из подписок отображаются раздельно;
- подписки импортируются и обновляются;
- при недоступности основного subscription server работает fallback через Яндекс Диск;
- повторный импорт той же подписки не создаёт дубликат;
- удалённое локально подключение возвращается после успешного обновления подписки, если оно всё ещё присутствует upstream;
- ручной пинг проходит через реально активный tunnel;
- release workflow использует cache `mobilecore`, если Go-код не изменён.

Текущий сетевой путь:

```text
Android applications
        ↓
Android VpnService / IPv4 TUN
        ↓
hev-socks5-tunnel
        ↓
локальный Xray SOCKS
        ↓
olcRTC SOCKS или standard outbound
        ↓
удалённый сервер / интернет
```

Текущий DNS-контракт для olcRTC-профилей:

```text
DNS приложений: 10.0.0.1:53 → Xray dns-out → 1.1.1.1:53 или global DNS
DNS carrier:     DNS из olcRTC-профиля → WBStream / Telemost carrier
```

## 3. Обязательные архитектурные ограничения

### 3.1. Что запрещено ломать или переписывать без отдельного обоснования

Следующие решения считаются рабочими инвариантами. Любое их изменение требует отдельного regression test и device acceptance:

- IPv4-only Android TUN и HEV, пока olcRTC SOCKS и серверный dial не поддерживают IPv6 end-to-end.
- Разделение tunnel DNS и carrier DNS.
- Tunnel DNS для olcRTC по умолчанию `1.1.1.1:53`.
- DNS из olcRTC-профиля используется только carrier-слоем.
- Пакет самого клиента всегда исключён из собственного VPN.
- Carrier sockets и configured resolver создаются через Android socket protector.
- TUN создаётся до запуска olcRTC/Xray/HEV.
- Сессия публикуется как `CONNECTED` только после успешной проверки реального datapath.
- Один последовательный владелец управляющего состояния start/stop/reconnect.
- Полный rollback частично запущенной native-сессии при ошибке.
- Сохранение desired-connected intent без сохранения секретов.
- Foreground `VpnService`, `systemExempted`, partial WakeLock, Quick Settings tile и state-aware notification.
- URL-only QR подписки с немедленным обновлением после импорта.
- Dedup подписок по URL.
- Atomic replacement профилей подписки и сохранение последнего рабочего набора при ошибке.
- Primary-first и Yandex mirror fallback.
- Ручной refresh без stale `ETag`/`Last-Modified`.
- Восстановление вручную удалённого subscription-профиля после refresh.
- Whitelist parsing, ограничения размера payload, bounded gzip и multipart QR.
- Android Keystore и AES-256-GCM для секретов.
- Отсутствие telemetry, analytics и автоматической отправки crash/log данных.
- Редактирование `CLAUDE.md` агентом запрещено; проектные знания хранятся в central vault.
- Cache `mobilecore` в GitHub Actions не меняется без реальной причины.

### 3.2. Что не делать в рамках этого цикла

- Не переписывать UI на Jetpack Compose.
- Не вводить DI-фреймворк или новую многослойную архитектуру только ради архитектуры.
- Не поддерживать произвольный Xray JSON, Clash YAML или sing-box JSON.
- Не добавлять новые VPN-протоколы без отдельного ТЗ.
- Не добавлять cloud sync аккаунтов и профилей.
- Не добавлять скрытые «тёплые» RTC-сессии в отключённом состоянии.
- Не подключаться параллельно к нескольким инстансам ради выбора самого быстрого.
- Не добавлять постоянный пользовательский auto-ping.
- Не снижать timeout только ради красивой цифры времени подключения: успешный readiness-check и так завершается раньше верхнего timeout.
- Не копировать Android-код или ресурсы Exclave. Допустимо сравнивать только поведение и контракты.

## 4. Целевая модель lifecycle подключения

### 4.1. Единственный источник истины

Источником истины о VPN остаётся `OlcrtcVpnService`, а не Fragment и не визуально выбранная карточка.

Service должен владеть:

- желаемым состоянием `desiredConnected`;
- активным profile reference;
- текущим `VpnState`;
- текущим этапом подключения;
- номером попытки reconnect;
- причиной последнего перехода;
- текущей physical network;
- active native session;
- результатом последней datapath-проверки.

UI только отображает эти данные и отправляет команды.

### 4.2. Состояния пользователя

Внутренние enum-значения не показываются пользователю напрямую.

| Внутреннее состояние | Текст в UI | Основное действие |
|---|---|---|
| `NO_PROFILE` | Текст состояния не показывать; отобразить empty state | Добавить подключение |
| `DISCONNECTED` | Не подключено | Подключить |
| `PREPARING` | Подготовка VPN | Отменить |
| `CONNECTING` | Подключение к `<имя>` | Отменить |
| `CONNECTED` | Подключено | Отключить |
| `RECONNECTING` | Восстановление соединения, попытка N | Отключить |
| `STOPPING` | Отключение | Кнопка временно недоступна |
| `ERROR` | Короткая понятная причина | Повторить |

### 4.3. Детальные этапы подключения

Service должен уметь публиковать безопасный `ConnectionStage` без секретов:

1. `LOAD_PROFILE` — загрузка профиля.
2. `WAIT_NETWORK` — ожидание физической сети.
3. `PREPARE_ASSETS` — подготовка Xray assets, только если она действительно нужна.
4. `CREATE_TUN` — создание Android VPN interface.
5. `START_CARRIER` — подключение WBStream/Telemost либо подготовка standard outbound.
6. `START_XRAY` — запуск Xray.
7. `START_HEV` — запуск HEV.
8. `VERIFY_DATAPATH` — реальная DNS/HTTP-проверка через tunnel.
9. `READY` — соединение готово.

Этап нужен для понятного интерфейса, локальной диагностики и измерения времени. Он не должен содержать room ID, client ID, auth token, ключи или URL с query string.

### 4.4. Отменяемый startup — приоритет P0

Текущий длительный `NativeSession.start()` выполняется в той же последовательной очереди, которая принимает stop, reconnect и network callbacks. Из-за этого команда «Отключить» во время медленного startup может ждать завершения readiness timeout.

Требуется:

- блокирующий native-startup не должен удерживать управляющую очередь;
- команда stop должна приниматься сразу в любом состоянии;
- startup должен иметь generation/attempt ID;
- результат устаревшей попытки после cancel игнорируется;
- cancel должен останавливать частично поднятые olcRTC, Xray, HEV и освобождать TUN;
- смена physical network во время startup должна отменять старую попытку и запускать новую на актуальной сети;
- одновременно может существовать только одна активная или запускаемая native-сессия.

Не требуется создавать сложный job scheduler. Достаточно сохранить один serialized control executor и вынести только блокирующую startup-работу в отдельный single worker с cancellation token.

### 4.5. Правила успешного подключения

Состояние `CONNECTED` разрешено публиковать только когда одновременно выполнено:

- TUN существует;
- HEV работает;
- Xray работает;
- olcRTC core работает для olcRTC-профиля;
- startup datapath probe успешно прошёл;
- попытка не была отменена;
- active network всё ещё совпадает с сетью, для которой создавалась сессия.

Локально открытый SOCKS port без успешного внешнего запроса не считается подключением.

## 5. Ускорение подключения

### 5.1. Сначала измерение, затем оптимизация

Перед изменением timeout или порядка запуска добавить локальные stage timings:

- время от нажатия до запуска foreground service;
- загрузка профиля и правил;
- подготовка GeoIP/GeoSite;
- создание TUN;
- olcRTC start и readiness;
- Xray start и readiness;
- HEV start;
- datapath verification;
- общее время до `CONNECTED`.

Timings пишутся только в redacted diagnostics и статистику разработки. Автоматически наружу они не отправляются.

### 5.2. Убрать лишнюю подготовку Geo assets

Сейчас `GeoAssetManager.prepare()` вызывается при каждом старте и может повторно считать SHA-256 крупных файлов и валидировать Xray config.

Требуется:

- при preset `ALL_VPN`, если custom rules не используют GeoIP/GeoSite, не подготавливать geo assets вообще;
- в рамках жизни service process успешно подготовленный каталог кешировать;
- сбрасывать cache после явного обновления geo assets, обновления приложения или обнаружения изменения файлов;
- не кешировать результат ошибки;
- сохранить проверку SHA-256 и rollback при обновлении assets.

### 5.3. Не выполнять скрытую тяжёлую работу после нажатия

- Профиль должен валидироваться при импорте и сохранении.
- Перед start выполняется короткая повторная проверка целостности, но не полный повторный parse/import.
- Список приложений не обновляется во время VPN startup.
- Stale subscriptions обновляются только после `CONNECTED` и не задерживают публикацию готового соединения.
- Проверка обновления приложения и geo assets не запускается автоматически при connect.

### 5.4. Не параллелить небезопасные native-этапы

Последовательность TUN → olcRTC → Xray → HEV → datapath verification сохраняется, пока benchmark не покажет конкретное узкое место и native API не гарантирует безопасный параллельный запуск.

Параллельный запуск глобальных singleton-компонентов `mobilecore` запрещён.

## 6. Быстрое и корректное отключение

### 6.1. Пользовательский результат

После нажатия «Отключить»:

- UI реагирует не позднее 100 мс;
- новая команда connect временно блокируется до освобождения TUN;
- Android route и DNS должны вернуться к обычной сети как можно раньше;
- notification и Quick Settings tile обновляются согласованно;
- native cleanup продолжается до полного завершения;
- повторный stop безопасен и идемпотентен.

### 6.2. Два измеряемых момента

Разделить метрики:

1. `routeReleased` — TUN больше не перехватывает пользовательский трафик.
2. `nativeStopped` — HEV, Xray и olcRTC полностью завершены.

Если текущий порядок закрытия не достигает целевых метрик, разрешается изменить порядок освобождения TUN и native-компонентов, но только с тестом на отсутствие FD/use-after-close ошибок.

### 6.3. Отключение во время startup/reconnect

Stop должен работать одинаково в `PREPARING`, `CONNECTING`, `CONNECTED`, `RECONNECTING` и `ERROR`:

- отменить pending reconnect;
- отменить startup attempt;
- запретить stale callback снова перевести service в `CONNECTED`;
- очистить `desiredConnected` только при явном пользовательском stop;
- закрыть logical statistics session один раз;
- снять WakeLock во всех ветках.

## 7. Reconnect и смена сети

### 7.1. Разделение ответственности

Нельзя запускать два конкурирующих reconnect-loop для одной причины.

- Внутренний carrier reconnect в mobilecore отвечает за кратковременный разрыв provider-сессии, если core умеет восстановиться без пересоздания Android VPN.
- `OlcrtcVpnService` выполняет full-session restart при смене physical network, завершении native-компонента или подтверждённой поломке всего datapath.
- Ошибка отдельного сайта, DNS-ответ `NXDOMAIN` или timeout одного пользовательского endpoint не является причиной full VPN reconnect.

Если существующих `isRunning()` и active probe недостаточно для различения состояний, native callback со статусом carrier добавляется только после подтверждённого измерениями пробела. Не добавлять callback заранее.

### 7.2. Классификация ошибок

Ошибки делятся минимум на три группы:

| Группа | Примеры | Поведение |
|---|---|---|
| Fatal configuration/auth | неверный ключ, auth failure, неподдерживаемая комбинация | остановить retry, показать исправляемую причину |
| Temporary network/provider | timeout, EOF, provider unavailable, потеря сети | сохранить desired state и reconnect |
| Local runtime | HEV/Xray завершился, TUN отозван | полный cleanup и контролируемый restart |

Raw exception остаётся в redacted diagnostics. В UI показывается короткий русский текст и действие: «Повторить», «Изменить подключение» или «Открыть диагностику».

### 7.3. Политика backoff

Рекомендуемый reconnect schedule при существующей physical network:

```text
0 с → 1 с → 2 с → 5 с → 10 с → 30 с → далее не чаще 60 с
```

Требования:

- первая попытка после реальной смены сети выполняется сразу;
- callback storm схлопывается коротким debounce 300–500 мс;
- при отсутствии physical network попытки не выполняются;
- появление сети немедленно сбрасывает ожидание и запускает попытку;
- fatal error останавливает цикл;
- transient error не очищает `desiredConnected` после трёх попыток;
- после длительной серии ошибок retry остаётся редким, чтобы не создавать battery/network spin;
- ручная кнопка «Повторить» сбрасывает backoff и запускает попытку сразу.

### 7.4. Wi-Fi ↔ мобильная сеть

При смене underlying network:

1. Новая eligible network фиксируется service.
2. Callback storm схлопывается.
3. Старая startup/session отменяется.
4. Старые native-компоненты полностью отсоединяются от предыдущей сети.
5. Создаётся новая сессия с `setUnderlyingNetworks(newNetwork)`.
6. `CONNECTED` возвращается только после нового datapath probe.

Старый network object нельзя повторно использовать после `onLost`.

### 7.5. Health monitoring

Сохранить два уровня проверки:

- частая локальная проверка `NativeSession.isRunning()`;
- редкая внешняя datapath-проверка через текущий Xray SOCKS.

Улучшения:

- health probe не должен делить очередь с ручным subscription refresh;
- одновременно выполняется не более одного probe;
- probe отменяется при stop, смене session generation или network change;
- одна случайная ошибка не должна немедленно уничтожать рабочую сессию; использовать две последовательные ошибки либо подтверждение от native runtime;
- успешный реальный трафик или успешный probe сбрасывает счётчик ошибок;
- timeout и интервал остаются консервативными и меняются только по device logs;
- probe не отображается пользователю как результат ручного пинга.

## 8. Фоновая работа

Уже работающие механизмы foreground service, WakeLock, `START_STICKY`, persisted intent и network callback сохраняются.

Дополнительные требования:

- пользователь может нажать «Подключить» и сразу свернуть приложение;
- закрытие Activity не отменяет startup;
- поворот экрана и смена вкладки не меняют service state;
- после возврата UI немедленно получает active profile и состояние через AIDL;
- после screen-off service не зависит от Activity process callbacks;
- после потери сети UI показывает «Ожидание сети» или «Восстановление», а не ложное «Подключено»;
- после восстановления сети трафик возвращается без ручного открытия приложения;
- WakeLock удерживается только пока desired VPN активен или выполняется cleanup;
- если Always-on VPN включён системой, клиент корректно восстанавливается после user unlock;
- отдельный boot receiver не добавляется, пока системного Always-on и `START_STICKY` достаточно.

## 9. Ручной пинг

### 9.1. Смысл функции

Пинг — это один реальный HTTP(S) запрос через активный Xray/olcRTC datapath, а не локальная проверка карточки и не TCP connect напрямую с телефона.

### 9.2. Поведение

- В состоянии `CONNECTED` пингуется только active profile независимо от визуальной selection.
- Результат устаревшей сессии не показывается после reconnect или stop.
- На время проверки кнопка блокируется и показывает progress.
- Успех `0 мс` запрещён; минимальное отображаемое значение — `1 мс`.
- Timeout, DNS error и отсутствие active session показываются разными сообщениями.
- Каждый ручной запуск создаёт новое соединение, без keep-alive reuse.
- Endpoint остаётся стабильным и не настраивается пользователем без отдельной необходимости.
- Постоянный auto-ping в карточках не добавляется.

### 9.3. Представление результата

Показывать:

```text
Пинг: 49 мс
Проверено только что
```

После ошибки:

```text
Не удалось проверить пинг
Тайм-аут соединения через VPN
```

## 10. Главный экран «Подключение»

### 10.1. Общая структура

Экран состоит из четырёх зон:

1. Верхняя панель с названием и компактным общим статусом.
2. Прокручиваемый список подключений.
3. Небольшая status/error область, появляющаяся только при полезном сообщении.
4. Закреплённая нижняя action area.

Рекомендуемый порядок:

```text
Подключение                         [общий статус]

Мои подключения
  [карточки]

Из подписок
  Подписка A
    [карточки]
  Подписка B
    [карточки]

[Подключить / Отключить / Отменить / Повторить]
[Добавить подключение] [Пинг]
```

Основные кнопки не должны уходить далеко за нижнюю границу при большом списке. Использовать прокручиваемый список и отдельную нижнюю action area, а не один бесконечный `ScrollView` со всеми действиями после списка.

### 10.2. Карточка подключения

Карточка показывает только необходимые данные:

- основная строка: пользовательское название;
- дополнительная строка только при необходимости: `WBStream`, `Telemost`, `VLESS`, `VMess` или `Trojan`;
- компактный статус `Подключено`, если карточка активна;
- edit и delete actions справа.

Не показывать в карточке room ID, server address, transport, UUID, DNS и другие технические поля.

### 10.3. Состояния карточки

- Активная карточка всегда ровно одна.
- Активная карточка использует нейтральный surface, 2 dp accent/contrast border, небольшую status mark и текст `Подключено`.
- Не использовать фиолетовую заливку всей карточки.
- Не использовать одновременно яркую заливку, толстую рамку и сильную elevation.
- Выбранная, но ещё не подключённая карточка имеет тонкий accent border.
- Остальные карточки сохраняют нормальный контраст; не приглушать весь текст через `alpha=0.55`.
- Во время подключения выбранная карточка показывает inline progress и текст этапа.
- Нажатие edit/delete не должно одновременно выбирать карточку.

### 10.4. Edit/delete actions

- Визуальный icon — 20–24 dp.
- Touch target — минимум 48×48 dp.
- Icon расположен строго по центру.
- Не использовать крупную outlined-кнопку вокруг каждой иконки.
- Delete использует destructive color только при нажатии/подтверждении, а не постоянно ярко-красную заливку.
- У каждой кнопки есть `contentDescription`.

### 10.5. Основная кнопка

Кнопка имеет высоту 56–60 dp и меняет подпись по состоянию:

- `Подключить`;
- `Отменить`;
- `Отключить`;
- `Повторить`;
- `Отключение…`.

Кнопки «Добавить подключение» и «Пинг» имеют одинаковую высоту, baseline текста и внутренние отступы. Текст не обрезается при русском языке и font scale 1.3.

### 10.6. Empty state

Если нет ни одного подключения:

- показать короткое объяснение;
- показать одну primary action «Добавить подключение»;
- не показывать disabled «Подключить» и «Пинг»;
- после первого импорта карточка появляется сразу без смены вкладки.

## 11. Добавление подключения

### 11.1. Bottom sheet

Оставить только два основных способа:

1. Сканировать QR-код.
2. Вставить из буфера обмена.

Каждый вариант оформляется как крупная action row:

- icon;
- заголовок;
- короткое описание;
- вся строка нажимаема.

Не использовать набор одинаковых серых текстовых кнопок.

### 11.2. Privacy

- Буфер обмена читается только после явного нажатия пользователя.
- Автоматическое чтение clipboard при открытии sheet запрещено.
- Импорт не подключает профиль автоматически без отдельного действия пользователя.
- Перед сохранением остаётся validation/preview.

### 11.3. Результат импорта

После успешного импорта:

- профиль автоматически сохраняется;
- появляется в правильной секции главного экрана;
- становится выбранным;
- показывается короткий snackbar/alert «Подключение сохранено»;
- кнопка «Подключить» становится доступной.

При повторном импорте duplicate identity обновляется существующая запись согласно текущему repository contract.

## 12. Редактор подключения

Текущий editor dialog с большим количеством полей заменить на полноэкранный editor либо высокий modal bottom sheet. Для сложного standard-профиля предпочтителен отдельный экран.

### 12.1. Структура

Поля группируются:

1. Основное: название, протокол/provider.
2. Подключение: адрес/room, port/transport.
3. Авторизация и безопасность.
4. DNS.
5. Дополнительные параметры.

### 12.2. Правила отображения

- Показывать только поля, применимые к выбранному provider/protocol/transport/security.
- Advanced-секция по умолчанию свёрнута.
- Для secret-полей есть reveal/hide, но нет автоматического копирования.
- Ошибки показываются inline у поля.
- Save недоступен при явно невалидной форме.
- При закрытии изменённой формы показывается подтверждение потери изменений.
- Активный профиль можно редактировать только с явным предупреждением, что изменения применятся после reconnect.
- После сохранения не выполнять reconnect автоматически без подтверждения пользователя.

### 12.3. Визуальный стиль

- Одна sticky action `Сохранить`.
- Secondary action `Отмена` в app bar.
- Отступы по 8 dp grid.
- Corner radius полей и контейнеров 8–10 dp.
- Не помещать форму внутрь нескольких вложенных карточек.

## 13. Экран «Подписки»

### 13.1. Карточка подписки

Показывать:

- название;
- количество подключений;
- время последнего успешного обновления;
- краткий статус источника: основной сервер / зеркало;
- последнюю ошибку только если она есть.

Actions:

- слева — `Обновить` с progress;
- справа — edit и delete либо одно overflow menu, если это визуально чище;
- вся карточка может открывать details.

### 13.2. Обновление

- Повторное нажатие во время refresh игнорируется.
- Progress привязан к конкретной подписке.
- Остальные карточки остаются интерактивными.
- По завершении показывается результат:

```text
Подписка обновлена
Добавлено: N
Удалено: N
Всего: N
Источник: основной сервер / зеркало
```

- При ошибке старые профили сохраняются.
- Результат `изменений нет` не считается ошибкой.
- Обновление списка подключений на главном экране происходит сразу.

### 13.3. Поведение duplicate subscription

Canonical URL остаётся идентификатором повторного импорта:

- существующая подписка обновляется;
- новая строка не создаётся;
- пользовательское имя не теряется без необходимости;
- старые профили сохраняются до успешного fetch;
- после успешного fetch применяется atomic replacement.

## 14. Настройки

### 14.1. Информационная архитектура

Не показывать одну длинную форму с большим количеством одинаковых кнопок.

Главный экран настроек разделить на компактные группы-строки:

- Маршрутизация;
- Приложения через VPN;
- DNS;
- Система и фон;
- Обновления;
- Диагностика;
- О приложении.

Каждая строка содержит title, текущее summary и chevron. Сложная настройка открывается на отдельном экране или в подходящем bottom sheet.

### 14.2. Унифицированные компоненты выбора

Использовать один принцип по всему приложению:

- binary choice — `MaterialSwitch`;
- 2–3 коротких взаимоисключающих варианта — segmented control либо компактная radio group;
- варианты с длинным описанием — bottom sheet с radio rows;
- большой список — отдельный searchable screen;
- опасное действие — отдельная destructive row с подтверждением.

Не использовать exposed dropdown для каждого выбора подряд.

### 14.3. DNS

В UI явно разделить:

- DNS трафика приложений;
- carrier DNS внутри olcRTC-профиля.

На глобальном экране показывать только tunnel DNS.

Под полем разместить пояснение:

> Используется для сайтов и приложений внутри VPN. DNS WBStream/Telemost задаётся в самом подключении.

Значение по умолчанию: `1.1.1.1:53`.

## 15. Выбор приложений для VPN-трафика

Текущий dialog создаёт checkbox для каждого приложения сразу. На устройствах с большим числом пакетов это медленно и визуально перегружено.

### 15.1. Новый экран

Использовать отдельный full-screen список:

- app icon;
- label;
- package name вторичной строкой;
- checkbox или trailing selection mark;
- строка поиска сверху;
- filter `Пользовательские / Системные`;
- счётчик выбранных;
- actions `Выбрать все`, `Очистить`;
- sticky `Сохранить`.

### 15.2. Производительность

- Сразу показывать Room cache.
- Обновлять PackageManager list в фоне.
- Не блокировать открытие экрана полным повторным scan.
- Не создавать все строки как вложенный `LinearLayout`.
- Для большого списка использовать `RecyclerView`/`ListAdapter` и стандартный diff.
- Search фильтрует локальный snapshot по label и package name.
- Результат ввода отображается не позднее 100 мс.
- Selection сохраняется при повороте экрана.
- Изменения записываются только после `Сохранить`.

### 15.3. Режимы

Режимы остаются прежними:

- Все приложения.
- Все, кроме выбранных.
- Только выбранные.

При `Только выбранные` и пустом списке Save запрещён с понятной ошибкой.

Пакет olcRTC Client остаётся исключённым независимо от режима.

## 16. Правила маршрутизации

Существующий порядок DNS → user rules → LAN → Geo → default сохраняется.

UI улучшить следующим образом:

- список правил вместо ввода строки `ACTION TYPE VALUE`;
- отдельные поля `Действие`, `Тип`, `Значение`;
- drag sorting не добавлять на первом этапе; достаточно кнопок вверх/вниз или стабильного порядка создания;
- enabled switch у каждого правила;
- inline validation domain/IP/CIDR;
- preview итогового поведения простым текстом;
- destructive delete с undo snackbar либо подтверждением.

Arbitrary raw Xray rule editor не добавлять.

## 17. Статистика

Сохранить локальный характер статистики и текущую Room history.

Улучшить представление:

- current session: профиль, длительность, сеть, upload/download, текущая скорость;
- today и month — две компактные summary cards;
- recent sessions — нормальный список с понятной причиной завершения;
- использовать theme semantic colors вместо hard-coded RGB;
- active session обновлять без полной перезагрузки экрана;
- очистку истории перенести в overflow/destructive action.

Графики, cloud history и детальная аналитика по доменам в этот цикл не входят.

Опционально после стабилизации lifecycle добавить к локальной сессии число reconnect и суммарное время восстановления. Для этого нужна отдельная Room migration и тест.

## 18. Диагностика и поддержка

### 18.1. Локальный connection timeline

Для каждой попытки сохранять redacted timeline:

```text
attempt started
profile loaded: 12 ms
network selected: wifi
TUN established: 40 ms
carrier ready: 3200 ms
Xray ready: 90 ms
HEV ready: 35 ms
datapath verified: 120 ms
connected: total 3497 ms
```

При ошибке фиксировать:

- safe error category;
- этап;
- elapsed time;
- network type;
- номер попытки;
- manual/network/health trigger.

### 18.2. Privacy

Не записывать:

- room ID;
- client ID;
- key;
- auth token;
- полный subscription URL с query;
- полный profile URI;
- список выбранных приложений.

Существующий redactor применяется перед показом, копированием и экспортом.

### 18.3. Экран диагностики

Показывать:

- app version;
- Xray version;
- olcRTC core version;
- текущее состояние;
- active profile type без secret identifiers;
- network type;
- последнюю safe error;
- действия `Открыть журнал`, `Копировать`, `Экспортировать`, `Сообщить о проблеме`.

Ничего не отправляется автоматически.

- Проверить light и dark theme.
- Проверить small screen и landscape.

## 21. Производительность и ресурсы

### 21.1. UI

- Первый frame главного экрана — без синхронного PackageManager scan.
- Cached profile/subscription list появляется без заметной задержки.
- Обновление списка не вызывает кратковременное исчезновение всех карточек.
- Для длинных списков использовать diff, а не `removeAllViews()` с полным повторным созданием.
- Background callback не обращается к уничтоженному ViewBinding.

### 21.2. VPN process

- Нет busy-loop при offline state.
- Одновременно не существует несколько scheduled reconnect для одной session generation.
- Одновременно не существует несколько health probes.
- Stop/cancel освобождает WakeLock, TUN и native resources.
- После 50 connect/disconnect циклов нет линейного роста threads, file descriptors или memory.
- Нативные библиотеки не пересобираются в CI, если их inputs не изменились.

## 22. Целевые метрики

Метрики проверяются на одном и том же устройстве, сети и сервере минимум в 10 последовательных запусках. Сохраняются median и p95.

### 22.1. Реакция интерфейса

- Ответ UI на connect/stop/ping: до 100 мс.
- Foreground notification после connect intent: до 500 мс.
- Открытие cached app selector: до 300 мс.
- Search update в app selector: до 100 мс.

### 22.2. Подключение на стабильном Wi-Fi

Целевые ориентиры, если provider отвечает нормально:

| Тип | Median | p95 |
|---|---:|---:|
| Standard VLESS/VMess/Trojan | ≤ 2 с | ≤ 5 с |
| Telemost olcRTC | ≤ 5 с | ≤ 15 с |
| WBStream olcRTC | ≤ 8 с | ≤ 30 с |

Если внешняя инфраструктура не позволяет достичь абсолютной цифры, обязательный критерий — улучшение median минимум на 20% относительно `v1.0.18` без ухудшения p95 и стабильности.

### 22.3. Отключение

- `routeReleased` median ≤ 300 мс, p95 ≤ 1 с.
- Полный native cleanup p95 ≤ 3 с.
- Cancel startup принимается ≤ 500 мс и не ждёт provider readiness timeout.

### 22.4. Reconnect

- Реакция на потерю active network: до 1 с.
- Первая попытка после появления новой сети: сразу после debounce.
- Восстановление Telemost на стабильной новой сети: median ≤ 5 с.
- Восстановление WBStream: median ≤ 10 с, если provider доступен.
- После 30 минут и 2 часов screen-off не требуется ручное открытие приложения.
- После wake рабочий трафик восстанавливается не позднее 10 с при доступной сети.

## 24. Автоматические тесты

Минимально добавить или расширить:

- cancellation generation: stale startup не публикует `CONNECTED`;
- stop во время каждого busy state;
- network change во время startup;
- transient failure не очищает desired intent;
- fatal failure останавливает retry;
- reconnect backoff и reset после появления сети;
- debounce callback storm;
- два последовательных health failures до full restart;
- health probe отменяется со старой session generation;
- geo assets не подготавливаются для `ALL_VPN`, когда не нужны;
- geo cache invalidation после update;
- единственная active connection card;
- ping result игнорируется после session change;
- profile/subscription list не теряется при refresh;
- app selector filtering и сохранение selection;
- accessibility content descriptions для icon-only actions;
- Room migration, если добавляются reconnect counters.

Существующие tests для DNS separation, IPv4-only HEV, subscription fallback, tombstones, QR, encryption и redaction сохраняются обязательными.

## 26. Порядок релизов

Не объединять в одном первом релизе:

- новую reconnect-модель;
- изменение DNS/routing;
- полный редизайн;
- Room migration;
- обновление mobilecore.

Рекомендуется:

1. Release A — lifecycle/cancellation/metrics, минимальные UI-тексты.
2. Release B — главный экран и app selector.
3. Release C — editors/settings/subscriptions/statistics.

После каждого release:

- GitHub Actions unit/instrumentation/build должны завершиться успешно;
- release APK собираются только workflow в репозитории;
- mobilecore пересобирается только при изменении его inputs;
- публикуются arm64-v8a, armeabi-v7a, universal и `SHA256SUMS.txt`;
- заполняются release notes;
- выполняется device acceptance;
- обновляется central vault `olcRTC/Project state.md`.

## 27. Definition of Done

Работа считается завершённой, когда:

- реализованный этап соответствует этому ТЗ;
- рабочий datapath `v1.0.18` не регрессировал;
- WBStream и Telemost проходят connect, traffic, screen-off и reconnect matrix;
- Chrome/GitHub/YouTube работают через VPN;
- tunnel DNS и carrier DNS остаются разделены;
- stop/cancel не ждёт полного provider timeout;
- active profile в UI всегда один;
- список подключений не исчезает при навигации или refresh;
- subscription primary/mirror/dedup/restore работают;
- ping создаёт реальный tunnel request и не показывает ложные `0 мс`;
- app selector быстро открывается и ищет по label/package;
- light/dark, RU/EN и font scale проверены;
- diagnostics не раскрывает секреты;
- CI не пересобирает mobilecore без изменения native inputs;
- signed release artifacts и checksum опубликованы;
- release notes и vault обновлены.

## 28. Краткий приоритет продукта

Если ресурсы ограничены, порядок строго такой:

1. Отменяемое подключение и быстрое отключение.
2. Стабильный reconnect и смена сети.
3. Измеримое ускорение startup без уменьшения надёжности.
4. Главный экран и понятные состояния.
5. Быстрый выбор приложений.
6. Редакторы и настройки.
7. Статистика и дополнительные улучшения.

Главный принцип следующего цикла: не переписывать уже рабочий VPN, а сделать его быстрее, управляемее, понятнее и визуально цельнее.
